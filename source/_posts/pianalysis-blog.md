# Pianalysis 项目日志：从单旋律到复调钢琴生成的实践复盘

## 0. 写在前面

这篇日志记录了我在 Pianalysis 项目中的完整技术实践：为什么选择 GPT-2 做符号化音乐生成、当前仓库的技术栈与代码实现、训练方式与参数设计、已得到的训练结果，以及目前暴露出的关键问题与下一步改进方向。

项目目标是一个典型的条件生成任务：输入单声部旋律线索，生成完整的复调钢琴音乐（Melody -> Full Polyphony）。当前代码与文档里既有可运行路径，也暴露出数据格式层面的核心瓶颈，因此这篇日志会把“做对了什么”和“哪里还没闭环”都写清楚。

---

## 1. 项目定位与任务定义

### 1.1 任务本质

Pianalysis 的核心是符号化音乐生成（Symbolic Music Generation），不是直接在音频波形上建模，而是在 token 序列上做自回归建模。这样做的优点是：

- 训练目标清晰，直接对应离散 token 的下一个预测。
- 可解释性相对更好，生成结果可回溯到事件级别（音符开关、时间步进等）。
- 训练与推理流程能直接复用 NLP 的 Transformer 生态。

### 1.2 当前仓库里的任务形态

当下任务是条件生成（Melody -> Polyphony）；但当前 `generate_from_scratch.py` 实现的是“无条件生成”（只用 BOS 起始 token 采样）。

这意味着项目现阶段更接近：

- 训练脚本支持在已有 token 序列上做语言建模。
- 推理脚本优先验证模型是否学到“可续写的音乐 token 分布”。
- 真正“可控条件生成”还需要在输入构造与推理接口上进一步补齐。

## 2. 技术栈总览

### 2.1 深度学习与建模

- PyTorch（训练与张量计算）
- Transformers（GPT2Config / GPT2LMHeadModel / Trainer）
- Tokenizers（WordLevel tokenizer 构建与保存）

### 2.2 数据与工程工具

- JSON 作为训练数据落盘格式
- NumPy / Pandas 用于数据处理扩展
- tqdm 用于进度可视化
- TensorBoard 用于训练日志监控

### 2.3 运行环境倾向

项目参数明显针对单卡消费级 GPU 进行调优（文档示例为 RTX 4070 Super 12GB），并在训练参数里显式考虑了 Windows 环境兼容性（`dataloader_num_workers=0`）。

## 3. 数据表示与 token 设计

### 3.1 当前训练输入接口

训练脚本直接读取每条样本中的 `training_sequence`，即“已经编码好的 token ID 列表”，并在 `Dataset` 层完成：

- 长度截断（超过 `max_length`）
- 右侧 padding
- attention mask 构造
- labels = input_ids（标准 Causal LM 训练方式）

这条路径的优势是训练链路简单，避免训练阶段再做复杂事件编码。

### 3.2 tokenizer 策略

仓库里 tokenizer 采用 WordLevel 方案，并约定了特殊 token：

- `<PAD>` = 0
- `<BOS>` = 1
- `<EOS>` = 2
- `<SEP>` = 3
- `<UNK>` = 4

在 `train_v2.py` 里，词表大小不是固定写死，而是通过扫描数据集中最大 token ID 自动推断，这一点很实用：当数据预处理更新后，不需要手工同步词表上限。

### 3.3 当前最大问题：数据语义不完整

README 已明确指出：当前数据分布仅覆盖很少 token，时间戳信息不完整，导致模型虽然能收敛 loss，但不代表学到了正确的音乐时序结构。

这是一个典型“训练指标好看但任务目标未达成”的情况。对于符号化音乐，时间维度是结构信息的核心，缺失时间 token 会直接破坏可演奏性与节奏合理性。

## 4. 模型实现细节（GPT-2 变体）

### 4.1 架构配置

当前模型使用轻量 GPT-2 配置：

- `n_positions = 2048`
- `n_embd = 512`
- `n_layer = 6`
- `n_head = 8`
- `dropout = 0.1`（resid/embd/attn）

这是一个参数量约 20M 的中小型自回归模型，适合先验证数据与建模闭环，再决定是否扩容。

### 4.2 为什么这组配置合理

- 层数与宽度平衡：6 层 + 512 hidden 对中等规模数据更稳，不会像超大模型那样对数据质量过于敏感。
- 上下文窗口 2048：为较长音乐片段建模留下空间。
- dropout 0.1：在数据质量不完美时，能一定程度缓解过拟合趋势。

### 4.3 训练框架选型

采用 Hugging Face `Trainer` 管理训练循环，优势包括：

- 训练、评估、保存检查点逻辑标准化
- 与 TensorBoard 集成成本低
- 可快速迭代超参数

同时，脚本里开启了 `load_best_model_at_end=True`，可在多轮训练后自动回到表现最佳的 checkpoint，工程实用性较高。

## 5. 训练方式与参数策略

### 5.1 数据切分

`train_v2.py` 采用 90/10 切分训练集与验证集（按样本顺序切分）。

一个可改进点是：如果样本在原始文件里存在风格聚簇或来源排序，顺序切分可能引入分布偏差。后续可考虑先打乱再切分，或按曲目 ID 分层切分。

### 5.2 优化参数

当前脚本核心训练参数：

- `max_length = 1024`
- `batch_size = 16`
- `epochs = 10`
- `learning_rate = 5e-4`
- `warmup_steps = 500`
- `weight_decay = 0.01`
- `save_steps = 1000`
- `logging_steps = 10`

此外，`fp16=torch.cuda.is_available()` 使其在支持混合精度的 GPU 上自动提升吞吐。

### 5.3 监控与早停意识

文档中记录了一个非常重要的现象：

- 前期 loss 从高位快速下降，学习正常。
- 后期 loss 继续极低，但伴随明显过拟合。

这说明项目已经具备了“识别过拟合并选择中间 checkpoint”的训练认知。下一步最好把这个经验固化成自动早停机制，而不是只靠人工观察。

## 6. 推理与生成实现

### 6.1 生成脚本特点

`generate_from_scratch.py` 采用从 BOS 起始的自回归采样，关键参数为：

- `temperature = 0.9`
- `top_k = 50`
- `top_p = 0.95`
- `num_return_sequences = 3`
- `max_length = 600`

这是较常见的“多样性与稳定性折中”设置，适合初步试听和分布检查。

### 6.2 输出处理

生成后会过滤 PAD/BOS/EOS 等特殊 token，并把纯 token 序列保存为 JSON，方便后续由外部解码器转换成 MIDI。

### 6.3 当前局限

因为数据端的时间 token 语义尚未修复，推理端即使采样策略合理，仍会输出结构不稳定的事件序列。这不是采样参数能根本解决的问题，必须回到数据预处理修正。

## 7. 项目中暴露的工程不一致点

在当前仓库里，可以观察到一些“文档/配置/代码并行演进”留下的不一致：

- `config.json` 中部分训练参数（如 batch size、split 比例、输出目录）与 `train_v2.py` 实际值不同。
- README 描述中既有“条件生成”的目标，也有“无条件生成脚本”的现实状态。
- 数据格式说明同时出现“复合 token 字符串示例”和“纯 token ID 序列训练输入”两套表述。

这类不一致在实验型项目非常常见，不是错误本身，但会影响协作效率。建议把“当前有效基线”与“目标设计草案”分区管理，避免相互覆盖。

## 8. 阶段性结果复盘

从现有记录看，项目已经完成了以下关键里程碑：

- 训练链路打通：数据加载 -> tokenizer -> GPT-2 -> Trainer -> checkpoint。
- 生成链路打通：加载 checkpoint -> 采样生成 -> 序列导出。
- 监控链路打通：TensorBoard 日志可用于 loss 追踪与过拟合观察。
- 最佳 checkpoint 的阶段性沉淀：例如 checkpoint-6000 被记录为相对更稳的选择。

同时，当前阻塞项也很明确：

- 数据时间信息缺失导致任务目标无法真正达成。
- 生成质量问题根因在数据语义，不在模型采样细节。

这说明项目已经从“工程可运行”进入“数据正确性驱动质量上限”的阶段。

## 9. 下一步技术路线（可执行版本）

### 9.1 第一优先级：修正数据预处理

目标是让每条序列都具有完整且一致的音乐事件语义，至少包括：

- 合理的时间推进 token（如 TIME_SHIFT 或绝对时间戳）
- NOTE_ON / NOTE_OFF 与 pitch 的合法配对
- 分轨或角色标识（melody/accompaniment）在训练序列中可被模型学习

### 9.2 第二优先级：重新训练并引入自动化停止策略

建议：

- 初始训练轮数先控制在 3-4 epoch
- 加入 early stopping 或更严格的验证集监控
- 保留最佳 checkpoint，而不是默认最后一轮

### 9.3 第三优先级：补齐条件生成接口

为了真正实现 Melody -> Polyphony，可在推理阶段支持：

- 输入前缀为给定旋律 token
- 只在伴奏相关 token 空间中进行受约束续写（或加 role-aware prompt）
- 输出端增加结构合法性检查（事件配对、时序单调性）

### 9.4 第四优先级：统一配置单一真源

将 `config.json` 与训练脚本参数对齐，或把训练脚本改为从配置文件读取，减少“参数漂移”导致的实验不可复现问题。

## 10. 总结

Pianalysis 目前最有价值的地方，不是“模型已经多强”，而是已经把一个完整的符号音乐生成系统骨架跑通了：

- 有明确任务定义
- 有可训练模型实现
- 有可复现实验流程
- 有阶段性结果与问题定位

下一阶段真正决定上限的是数据质量，尤其是时间结构与事件语义的一致性。只要把这一步补齐，现有 GPT-2 管线就具备继续迭代到更高质量结果的基础。

这也是这次实践最大的经验：在生成任务里，模型能力是必要条件，数据表达才是充分条件。
